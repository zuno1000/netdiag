# 通信診断 — つながらない理由がわかるアプリ

「ネットが遅い」「つながらない」の原因をその場で調べて、
**道路の渋滞にたとえた絵**でわかりやすく表示するWebアプリです。
サーバを持たない完全無料構成（GitHub Pages + 端末内処理のみ）で動きます。

## 使い方（利用者向け）

### はじめに

1. ブラウザで公開URLを開く（例: `https://<ユーザ名>.github.io/netdiag/`）
2. **iPhone**: Safariの共有ボタン →「ホーム画面に追加」。以降はアプリのように起動でき、
   **一度開いた後は電波が全く無くても起動します**（診断アプリが「圏外で開けない」ことはありません）

### 診断する

- **「診断する」ボタン**を押すと約5〜10秒で結果が出ます。計測中は進捗バーに
  「今なにを計測しているか」「残り約何秒か」が表示されます
- 結果は3つの形で表示されます:
  - **道路の絵**: スマホ→入口（Wi-Fi・基地局）→案内所（DNS）→インターネット→相手サーバ
    のどこが詰まっているかを、車の流れと色で表現
  - **ことばの説明**: 原因と「どうすればよいか」の提案
  - **用途別の目安**: 💬メッセージ / 🌐Web・地図 / 🎬動画 / 📞ビデオ通話 が
    今の回線で使えそうか（◯△✕）

### 結果に応じてできること

- **詳しく調べる**（「遅い/不安定」の時に表示）: 追加で最大約2MB通信して、
  「誰かが回線を使っている」「速度制限かも」「相手サーバ側の問題」などまで切り分けます
- **回復を見張る**（不調の時に表示）: 30秒ごとに軽く確認し、直ったらお知らせします。
  画面を開いたままにしてください
- **回線を切り替えて比較**: Wi-Fi⇄モバイル回線を切り替えて2回計測し、
  どちらが快適かを表で比較・提案します
- **画像で共有**: 診断結果を1枚の画像にして、家族やサポート窓口に送れます

### 記録について

- **場所タグ**: 診断する前に「自宅」「電車・移動中」などを選ぶと記録に付きます
- **履歴グラフ**: 過去の応答時間の推移を折れ線で確認できます
- **CSVエクスポート**: すべての記録をExcelで開ける形式で保存できます
- 記録は**この端末の中にだけ**保存されます。外部には一切送信されません

## プライバシーとセキュリティ

- サーバを持たない静的サイトで、利用者のデータを収集・送信する仕組みは無い
- 診断記録は端末のlocalStorageのみに保存され、外部には送信されない
- 計測時にCloudflare・Google・jsDelivrへ小さなHTTPリクエストを送る
  （通常のWeb閲覧と同程度。相手には他のWebアクセス同様にIPアドレスが見える）
- 配信はGitHub PagesのHTTPS。リポジトリは公開だが、書き込めるのは所有者のみ
- ユーザ実験で使う場合は、上記（外部送信なし）を説明文書に明記できる

## ローカルでの動かし方（Windows）

```
cd netdiag-pwa
python -m http.server 8000
```

ブラウザで http://localhost:8000 を開く（`file://` では動かない）。
開発中は DevTools → Application → Service Workers →「Update on reload」を有効にする。

## 無料公開（GitHub Pages）

1. GitHubで新規リポジトリを作成（Public）
2. このフォルダの中身をリポジトリ直下にアップロード
3. Settings → Pages → Branch を「main / (root)」にして保存
4. 数分後 `https://<ユーザ名>.github.io/<リポジトリ名>/` で公開される

## 維持費が0円である根拠

- ホスティング: GitHub Pages（無料・静的サイト）
- サーバサイド処理: なし（診断・判定・描画・記録はすべて端末内で完結）
- プローブ先: Cloudflare等の公開エンドポイント（無料）。仕様変更に備えるなら
  Cloudflare Workers無料枠で自前エンドポイントを立て、`probeUrl` を差し替える:

```js
export default {
  async fetch() {
    return new Response("ok", {
      headers: {
        "Access-Control-Allow-Origin": "*",
        "Timing-Allow-Origin": "*",
        "Cache-Control": "no-store",
      },
    });
  },
};
```

## 詳細診断（二段構え）

初回診断が「遅い/不安定」の時だけ「詳しく調べる」ボタンが現れる。
追加で最大約2MB・約10秒の計測を行い、「遅い」を4方向に切り分ける:

| 判定 | 根拠 |
|---|---|
| 回線が飽和（誰かが使用中） | 負荷をかけた時だけRTTが3倍以上に膨らむ（バッファブロート） |
| 速度制限の可能性（ギガ切れ等） | 実効速度が300kbps未満 |
| 自分の回線全体が遅い / 単に遠い | 運営主体の異なる3計測先が全て遅い（帯域が太ければ「遠い」） |
| 相手サーバ側が遅い | 特定の計測先だけ遅い、または接続は速いのに応答待ちだけ長い |

しきい値は `probe-engine.js` の `THRESHOLDS` で調整できる。
判定ロジックは `NetDiag._internals.diagnoseDeep` としてエクスポートしてあり、
Node.jsで単体テストできる。

## テスト方法（悪環境の再現）

Chrome DevTools → Network タブ → スロットリングで「Slow 3G」「Offline」等を選び、
各診断分岐が出ることを確認する。損失・揺らぎの再現には
[clumsy](https://jagt.github.io/clumsy/)（無料）でパケット損失・遅延を注入する。

## ブラウザ版の既知の制約（論文の限界節に書ける内容）

- 回線種別（Wi-Fi/セルラー、5G/LTE）と電波強度は取得不可（iOS Safari）
- DNS失敗とTCP失敗の区別は不可。内訳は「所要時間」として取得（Timing-Allow-Origin必須）
- 2回目以降はTCP接続が再利用され、DNS/TCP/TLS内訳が出ないことがある
- `navigator.onLine` は「LANにつながっているか」しか見ておらず過信できない
- キャプティブポータル検出は間接推定
- 回復ウォッチは画面を開いている間のみ動作（バックグラウンド通知は不可）
